# DockStart 全功能扩展计划书

> 目标：在不牺牲 DockStart 现有“现代化、简洁、引导清晰、开箱即用”体验的前提下，逐步补齐 AutoDock Vina 已有但 DockStart 尚未完整支持的功能，并建立可持续扩展的对接协议架构。

- 项目仓库：<https://github.com/xuxinxi14/DockStart>
- 当前基线：DockStart v0.11.0
- 当前内置核心：AutoDock Vina 1.2.7
- 计划性质：产品路线图 + 技术架构方案 + 开发实施计划 + 验收标准
- 实施原则：分阶段发布，不进行一次性大重写

---

## 1. 项目背景

DockStart 当前已经形成较完整的标准分子对接流程：

1. 创建或打开项目；
2. 导入或准备受体、配体；
3. 设置对接 Box；
4. 设置 Vina/Vinardo 常用参数；
5. 运行前检查；
6. 执行 AutoDock Vina；
7. 查看构象、分数和 RMSD；
8. 导出 CSV 与 Markdown 实验记录。

现阶段 DockStart 的主要优势不是修改对接算法，而是把分散的命令行工具、文件格式、参数设置和结果归档整理成清晰、可追踪、适合初学者的本地工作流。

下一阶段不应简单地把 AutoDock Vina 的全部命令行参数堆进一个“高级设置”页面，而应把不同科学用途拆分成明确的“对接协议”，为每种协议提供独立的输入要求、准备流程、参数、运行前检查、结果解析和报告模板。

---

## 2. 总体目标

### 2.1 产品目标

将 DockStart 从“标准刚性单配体对接工作台”逐步升级为：

> 面向 AutoDock Vina 生态的现代化、本地化、可追溯分子对接工作台。

完成后应覆盖：

- 标准刚性单配体对接；
- Vina 与 Vinardo；
- 高级搜索参数；
- `score_only` 现有姿势评分；
- `local_only` 局部优化；
- 批量虚拟筛选；
- 柔性侧链对接；
- 大环配体工作流；
- AutoDock4 affinity maps；
- AutoDock4 评分；
- 多配体同时对接；
- AD4Zn 专用协议；
- 水合对接；
- 外部 maps 导入、生成与复用；
- 完整任务队列、结果汇总和审计记录。

### 2.2 非目标

本计划不包含：

- 修改 AutoDock Vina 核心算法；
- 自研新的 scoring function；
- 将 docking score 描述为真实结合自由能；
- 自动生成药效、安全性或临床结论；
- 分子动力学模拟；
- QM/MM；
- 自动论文生成；
- 无人工确认的全自动受体修复；
- 无依据的自动质子化、电荷或结合位点判断。

### 2.3 成功标准

1. 初学者仍能在默认流程中完成标准对接，不被高级功能干扰。
2. 高级用户可以选择明确的协议，而不是手动拼装命令行参数。
3. 所有运行均保存输入、配置、命令、工具版本、日志、输出和哈希。
4. 每种协议都有独立的科学边界说明与失败提示。
5. Basic 安装包保持精简；Assisted 安装包继续开箱即用。
6. 复杂功能采用逐级披露，不破坏现有现代化界面。
7. 项目文件具有向前迁移、向后保护和异常恢复能力。

## 3. 产品设计原则

### 3.1 默认流程必须保持简单

默认首页和新建项目入口只突出三类常用场景：

- 标准小分子对接；
- 批量虚拟筛选；
- 现有姿势评分与优化。

柔性侧链、AutoDock4、AD4Zn、水合对接等高级协议放入“更多协议”，不在首页同时展示大量入口。

默认标准对接仍只要求用户理解：

- 受体；
- 配体；
- Box；
- 搜索彻底程度；
- 构象数量；
- 开始运行。

### 3.2 使用“协议”而不是“参数堆叠”

不建议继续把全部功能加入现有 `VinaSettings`。建议建立协议类型：

```text
rigid_single
batch_screening
score_only
local_only
flexible_sidechain
macrocycle
ad4_maps
multiple_ligands
ad4zn
hydrated
```

每种协议拥有：

- 独立输入模型；
- 独立工具依赖；
- 独立准备流程；
- 独立参数模型；
- 独立前置检查；
- 独立命令生成器；
- 独立输出解析器；
- 独立报告章节；
- 独立科学风险提示。

### 3.3 逐级披露

每个协议的参数页面分成三层：

#### 基础参数

只显示多数用户必须设置的内容。

#### 推荐设置

提供可解释预设：

- 快速预览；
- 标准；
- 深度搜索；
- 自定义。

#### 专家参数

折叠显示：

- `min_rmsd`；
- `max_evals`；
- `spacing`；
- `verbosity`；
- `no_refine`；
- scoring weights 等。

专家参数必须显示默认值、单位、作用、风险、可比性影响，并提供“一键恢复默认值”。

### 3.4 自动化只能代替机械步骤，不能代替科学判断

允许自动完成：

- 路径管理；
- 文件复制；
- 配置生成；
- 工具检测；
- 格式检查；
- 任务排队；
- 日志归档；
- 结果汇总；
- 哈希记录。

必须要求人工确认：

- 受体链选择；
- 缺失残基处理；
- 水分子、金属、辅因子保留；
- 质子化状态；
- 配体形式电荷；
- 互变异构体；
- 柔性残基选择；
- 大环断环位置；
- Zn 配位环境；
- Box 是否覆盖真实研究位点。

### 3.5 保持本地优先

除主动下载 RCSB/PubChem 数据外：

- 结构准备在本机完成；
- 对接在本机完成；
- 项目数据在本机保存；
- 诊断在本机生成；
- 不上传私有结构；
- 不依赖云端账户才能运行。

## 4. 总体架构方案

### 4.1 协议注册中心

新增统一协议接口：

```python
class DockingProtocol:
    protocol_id: str
    display_name: str
    maturity: str
    required_tools: list[str]

    def validate_inputs(...)
    def prepare_inputs(...)
    def build_config(...)
    def build_command(...)
    def preflight(...)
    def parse_outputs(...)
    def build_report(...)
```

协议注册中心负责：

- 返回当前可用协议；
- 判断 Basic/Assisted 是否满足依赖；
- 返回协议说明；
- 返回表单 schema；
- 调用对应运行适配器；
- 统一保存审计信息。

### 4.2 项目数据模型升级

建议将当前项目 schema 升级为 v2：

```json
{
  "schema_version": 2,
  "project_name": "...",
  "protocol": {
    "id": "rigid_single",
    "version": 1,
    "maturity": "stable"
  },
  "inputs": {},
  "preparation": {},
  "search_space": {},
  "parameters": {},
  "runs": []
}
```

要求：

- 保留 v1 项目迁移；
- 旧项目默认映射为 `rigid_single`；
- 迁移前自动备份；
- 未知字段继续保留；
- 高版本项目不得被低版本程序强行改写；
- 每次运行保存协议版本，避免以后逻辑变化影响历史重现。

### 4.3 统一运行任务模型

所有协议使用统一状态：

```text
created
preparing
queued
running
cancelling
cancelled
finished
failed
interrupted
recoverable
```

任务记录至少包含：

- task_id / run_id；
- protocol_id；
- 输入快照；
- 配置快照；
- 命令数组；
- 工具版本与哈希；
- 开始、结束时间；
- PID 与进程身份；
- stdout/stderr/log；
- 输出文件；
- 解析结果；
- 错误代码；
- 重试来源；
- 取消原因；
- 恢复状态。

### 4.4 统一结果模型

```text
ligand_id
pose_id
affinity
rmsd_lb
rmsd_ub
status
warning_count
output_file
protocol
scoring_function
```

协议可附加扩展字段，例如柔性残基、大环断环键、maps 信息、Zn 配位参数、水合状态和多配体成员。

# 阶段 0：架构准备与基线冻结

## 目标

先建立协议化、任务化和可迁移的数据基础，避免后续反复重构。

## 工作内容

### 0.1 冻结稳定基线

- 标记当前稳定版本；
- 保存 Basic 与 Assisted 安装态验收结果；
- 固定标准单配体基准案例；
- 固定 Vina 与 Vinardo 回归输出容差；
- 记录当前项目 schema；
- 建立升级前兼容测试。

### 0.2 引入协议抽象

- 建立 `DockingProtocol` 接口；
- 将现有标准对接迁移为 `rigid_single`；
- 保持现有前端流程和用户体验基本不变；
- 本阶段不增加新科学功能。

### 0.3 升级项目 schema

- 设计 schema v2；
- 实现 v1 → v2 迁移；
- 增加迁移备份；
- 添加协议版本；
- 添加运行模式字段；
- 添加扩展参数容器；
- 增加损坏项目、未知字段和高版本项目测试。

### 0.4 统一错误系统

错误代码按领域分组：

```text
PROJECT_*
INPUT_*
PREPARATION_*
PROTOCOL_*
MAPS_*
AUTOGRID_*
VINA_*
BATCH_*
FLEX_*
MACROCYCLE_*
AD4ZN_*
HYDRATED_*
RESULT_*
REPORT_*
```

错误提示统一包含：用户可读说明、原始错误、建议操作、影响范围、是否阻塞和可跳转页面。

## UI 要求

仅增加：

- 项目协议标记；
- 旧项目迁移提示；
- 兼容性说明；
- “查看迁移备份”入口。

## 验收标准

- 旧项目可以无损打开；
- 旧运行记录保持可查看；
- 标准对接结果与升级前一致；
- Basic/Assisted 均通过安装态测试；
- 未知字段不丢失；
- 迁移失败时原项目不被改写；
- 所有测试、构建和 `cargo check` 通过。

# 阶段 1：高级参数与轻量运行模式

## 目标

优先加入实现成本较低、科研用途明确的 Vina 原生能力。

## 功能范围

### 1.1 高级搜索参数

新增：

- `min_rmsd`；
- `max_evals`；
- `spacing`；
- `verbosity`。

可选新增：

- `no_refine`；
- `force_even_voxels`。

默认值与 Vina 保持一致。

### 1.2 `score_only`

建立独立协议：对现有受体—配体姿势进行评分，不进行全局搜索。

支持：

- 普通 Box；
- 可选 autobox；
- 可选 unbound energy；
- 可选 no_refine；
- 评分项解析；
- 输出评分报告。

### 1.3 `local_only`

建立独立协议：从输入姿势附近执行局部优化。

支持：

- 输入姿势预览；
- 优化前后结构对比；
- 优化前后分数对比；
- 输出优化后的 PDBQT；
- RMSD 或重原子位移说明。

### 1.4 `randomize_only`

作为专家工具，不作为主要首页入口。明确提示随机化姿势不是 docking 结果。

## UI 设计

新建项目页增加“姿势评分与优化”，进入后再选择：

- 只评分；
- 局部优化；
- 随机化姿势。

标准对接参数页保持简洁，专家参数默认折叠。

## 验收标准

- 默认标准流程不增加必填项；
- 新参数正确写入运行快照；
- `score_only` 不生成虚假的 pose 排名表；
- `local_only` 正确保存优化后结构；
- 报告明确区分全局对接、评分和局部优化；
- 不同 scoring function 的分值不横向混排。

# 阶段 2：批量虚拟筛选

## 目标

建立 DockStart 最重要的扩展能力：单受体、多配体的本地批量筛选工作流。

## 功能范围

### 2.1 配体库导入

支持：

- 选择多个 PDBQT；
- 导入包含 PDBQT 的目录；
- 导入 SDF 多分子文件；
- SDF/MOL 批量准备；
- 识别重复名称；
- 生成稳定 ligand_id；
- 保存原始文件与准备后文件映射。

### 2.2 配体表格

默认字段：

- 名称；
- ligand_id；
- 来源；
- 准备状态；
- 形式电荷提示；
- 重原子数；
- 可旋转键；
- 错误状态；
- 是否加入运行；
- 最新结果。

支持搜索、过滤、排序、批量选择、失败项重试和失败清单导出。

### 2.3 任务队列

实现：

- 队列生成；
- 顺序执行；
- 可配置并发；
- CPU 总量上限；
- 全局取消和单项取消；
- 中断恢复；
- 失败重试；
- 跳过已成功任务；
- 防止重复输出覆盖。

初期建议使用 DockStart 自己的队列逐个调用 Vina，而不是完全依赖 `--batch`，以便实现精细进度、单项重试、独立日志和恢复。后续可增加“原生 Vina batch 模式”作为性能选项。

### 2.4 结果汇总

支持：

- Top N；
- 每个配体最佳 affinity；
- 每个配体最佳 pose；
- 状态筛选；
- 错误筛选；
- CSV 导出；
- SDF 汇总；
- PDBQT 打包；
- 按分数排序；
- 点击配体打开 3D pose；
- 选择多个结果并排比较。

### 2.5 批量报告

包含配体总数、成功/失败/取消数量、评分函数、Box、参数、工具版本、运行耗时、Top N、失败原因统计与科学免责声明。

## UI 设计

采用三层页面：

1. 配体库；
2. 任务队列；
3. 筛选结果。

首页只显示一个“批量虚拟筛选”卡片，不展示复杂队列参数。数据表格默认只显示关键列，其他列通过“列设置”打开。

## 验收标准

- 100、1000 个配体项目可正常建立；
- 单个配体失败不阻塞整个队列；
- 中断后可识别已完成项；
- 不覆盖历史结果；
- 总 CPU 不超过用户上限；
- 每个配体有独立输入、配置、日志和结果；
- Top N 与原始结果一致；
- 大型项目不一次性把全部结构内容载入前端内存。

# 阶段 3：柔性侧链对接

## 目标

支持受体有限侧链柔性，同时保持操作可理解、可追踪。

## 功能范围

### 3.1 柔性残基选择

- 在 3D 视图点击残基；
- 按链与残基号搜索；
- 显示距参考配体或 Box 中心的距离；
- 限制选择数量；
- 排除不完整残基；
- 明确显示当前选中残基。

### 3.2 rigid/flex 受体准备

- 生成 rigid PDBQT；
- 生成 flex PDBQT；
- 保存链、残基号、残基名称；
- 保存准备命令；
- 验证 rigid/flex 文件一致性；
- 检查重复原子和缺失原子；
- 记录准备工具版本。

### 3.3 参数与运行

增加柔性残基数量、搜索彻底程度建议、CPU 与运行时间提示、Box 覆盖检查和自由度风险提示。

### 3.4 结果显示

3D Viewer 支持：

- 刚性受体；
- 柔性侧链；
- 配体 pose；
- 柔性侧链前后构象比较；
- 按残基高亮；
- 结果中的柔性残基列表。

## UI 设计

采用向导：导入结构 → 检查结构 → 选择柔性残基 → 生成 rigid/flex → 设置 Box → 设置参数 → 运行。

## 验收标准

- 官方柔性对接案例可复现；
- rigid 与 flex 原子划分无重复；
- 不完整残基被阻止或明确警告；
- 柔性残基写入报告；
- 普通刚性项目不受影响；
- 结果页面能区分配体变化和侧链变化。

# 阶段 4：AutoDock4 maps 基础设施

## 目标

建立 AD4、AD4Zn 和水合对接共同依赖的 maps 基础设施。

## 功能范围

### 4.1 AutoGrid4 工具链

- 完成许可证与分发审查；
- 确定 Basic/Assisted 分发策略；
- 工具检测；
- 版本记录；
- 哈希校验；
- 安装态验收。

### 4.2 GPF 生成

支持受体、原子类型、grid center、grid points、spacing、参数文件、maps 前缀、AutoGrid 日志和错误解析。

### 4.3 maps 管理

- 生成 maps；
- 导入已有 maps；
- 验证 maps 完整性；
- 识别缺失原子类型；
- 保存 maps manifest；
- 记录 maps 与受体的哈希关联；
- 复用 maps；
- 受体变化后旧 maps 自动失效。

### 4.4 AutoDock4 评分协议

新增独立协议 `AutoDock4 (maps)`：

- 不与 Vina/Vinardo 放在普通评分函数下拉框中；
- 必须先准备或导入 maps；
- 分数结果独立展示；
- 报告明确注明 AD4；
- 禁止把 AD4 分数与 Vina/Vinardo 直接比较。

## 验收标准

- 官方 AD4 案例可复现；
- maps 缺失时禁止运行；
- 原子类型不匹配时给出明确说明；
- 受体变更后旧 maps 自动失效；
- maps manifest 完整；
- AutoGrid 错误不会被误报为 Vina 错误。

# 阶段 5：大环配体正式支持

## 目标

把 Meeko 可能已有的底层大环准备能力升级为正式、可审计的 DockStart 协议。

## 功能范围

### 5.1 大环检测

- 检测环尺寸；
- 检测可处理的大环；
- 标记潜在断环键；
- 显示 Meeko 支持状态；
- 不支持时给出明确原因。

### 5.2 断环选择

- 自动建议；
- 3D 显示候选键；
- 人工确认；
- 保存最终断环位置；
- 记录原子编号与键类型；
- 恢复默认建议。

### 5.3 PDBQT 准备验证

检查 glue atom、断环标记、环原子对应关系、输出完整性、准备日志和 Meeko 版本。

### 5.4 结果解释

报告明确说明断环与伪原子方法、环闭合检查要求、分数解释限制和断环位置影响。

## UI 设计

普通配体准备检测到大环时，显示“检测到大环配体”与“进入大环准备”按钮，不直接失败，也不静默当作普通配体处理。

## 验收标准

- 官方大环案例可复现；
- 断环位置完整记录；
- glue atom 可验证；
- 不支持的大环不会被静默接受；
- 普通非大环流程不增加额外步骤。

# 阶段 7：AD4Zn 专用协议

- 提醒确认辅因子、水和配位残基；

## 验收标准

- 非 Zn 金属不会误进入协议；

# 阶段 9：评分权重与开发者接口

## 目标

为专家和开发者提供可控扩展能力，但不干扰普通用户。

## 功能范围

### 9.1 自定义评分权重

支持 Vina、Vinardo、AD4 和 macrocycle glue 权重。

限制：

- 默认关闭；
- 进入实验性专家模式；
- 修改后项目和报告显著标记；
- 自定义分数不与标准权重结果混排；
- 一键恢复官方默认值。

### 9.2 Vina Python API

可选增加：

- Python API adapter；
- notebook 示例；
- 批量分析接口；
- 插件开发接口；
- 只读项目结果 API；
- 不允许插件绕过项目安全边界。

### 9.3 协议插件机制

外部插件可注册输入检查、准备步骤、命令生成、结果解析和报告扩展。插件必须声明工具依赖、许可证、成熟度和数据访问范围，不能默认联网或静默修改原始项目。

## 9. 风险与控制措施

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 功能堆叠导致界面复杂 | 初学者无法使用 | 协议分类、逐级披露、默认流程不变 |
| 项目 schema 频繁变化 | 历史项目损坏 | 版本化、迁移备份、未知字段保留 |
| 批量任务占满 CPU | 系统卡顿 | 总 CPU 上限、并发限制、暂停/取消 |
| 配体库规模过大 | 前端卡顿 | 分页、虚拟列表、延迟加载 |
| AD4 maps 错配受体 | 科学结果无效 | 哈希绑定、manifest、变更自动失效 |
| 柔性残基选择错误 | 搜索空间失控 | 数量限制、结构检查、显式确认 |
| 大环被误当普通配体 | 结果不可解释 | 自动检测、专用协议、断环记录 |
| 金属体系被过度泛化 | 误用 AD4Zn | 仅限 Zn、人工确认、专用提示 |
| 自定义权重破坏可比性 | 分数误读 | 实验性标签、独立结果、禁止混排 |
| 第三方许可证问题 | 无法合法分发 | 组件级审查、源码材料、notices |
| 运行被中断 | 项目状态不一致 | 原子写入、任务状态机、恢复检查 |
| 高级功能测试不足 | 错误进入 Stable | maturity 分级、实验性隔离、基准门禁 |

## 10. 成熟度管理

每种协议标记：

```text
stable
beta
experimental
developer
```

### Stable

完整 UI、文档、自动化测试、安装态测试、科学回归、可恢复运行和清晰报告。

### Beta

主要流程可用，已知限制明确，不进入默认推荐。

### Experimental

独立开关，不承诺结果稳定，不进入默认首页，不与 Stable 结果混排。

### Developer

仅用于调试或插件开发，不面向普通用户。

## 17. 总结

DockStart 下一阶段的关键不是“把 Vina 的所有参数全部搬到界面”，而是：

> 用清晰的协议、可靠的准备流程、严格的运行前检查、可恢复的任务系统和可追溯的结果记录，把 Vina 的高级能力整理成真正可用的现代化科研工作流。

整个扩展过程应坚持：

- 默认流程简单；
- 高级能力分层；
- 本地优先；
- 开箱即用；
- 科学边界明确；
- 历史项目安全；
- 每阶段可发布；
- 每项功能可验收；
- 不用表面完整代替实际可靠。

最终目标是让 DockStart 从当前的标准对接工作台，逐步成长为覆盖 AutoDock Vina 主要能力、适合初学者与科研用户长期使用的一站式本地分子对接平台。
